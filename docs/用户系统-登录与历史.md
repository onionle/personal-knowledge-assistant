# 用户系统 · 登录 + 会话历史

> 在三种实现（LangChain Python / LangChain4j / Spring AI）之上加一层用户系统：
> 注册登录、把每段对话历史存进数据库、查看/重命名/删除历史会话，并把历史作为多轮对话的上下文。
> 三个后端**共用同一个 MySQL 库和同一个 JWT 密钥**，所以账号互通、token 互认。

---

## 一、要解决什么

前面的对话只有进程内记忆——重启就没了，也分不清是谁的。用户系统补上：

- **登录**：注册/登录，拿到 JWT，之后请求带上它就知道"你是谁"。
- **历史**：登录后每轮问答落库；可列出"我的会话"、点开看完整记录。
- **管理**：会话可**重命名**、**删除**（逻辑删除，不真删行）。
- **上下文**：多轮对话的上下文从数据库取——重启、换机器也能接着聊。

> 匿名（不登录）依然能用：对话走进程内记忆、历史存在浏览器本地，只是不入库。

---

## 二、数据库（MySQL，3 张表）

建表脚本：`db/schema.sql`（`mysql -u root -p < db/schema.sql`）。

| 表 | 作用 | 关键字段 |
|----|------|---------|
| `users` | 用户 | `username` 唯一、`password_hash`（bcrypt） |
| `conversations` | 会话 | `id`=UUID（即 conversation_id）、`user_id`、`title`、`backend`、`status`（0正常/1已删除）、`updated_at` |
| `messages` | 消息 | `conversation_id`、`role`（user/assistant）、`content` |

- 外键级联删除；`conversations(user_id, status, updated_at)` 索引支持"按用户列最近未删除会话"。
- **逻辑删除**：删除只把 `status` 置 1，行还在（可恢复/审计），列表查询过滤 `status=0`。

---

## 三、鉴权（JWT）

- 注册时密码用 **bcrypt** 哈希入库（绝不存明文）。
- 登录成功签发 **JWT（HS256）**，载荷含 `sub`=用户id、`username`、`exp`。
- 三个后端用**同一个 `JWT_SECRET`**（`.env`）+ 同样的 HS256，所以**同一个 token 三个实现都认**；
  bcrypt 也通用，所以**一个账号三个后端都能登**。
- 请求带 `Authorization: Bearer <token>`。聊天接口用"可选鉴权"：带 token 就落库、不带就匿名。

---

## 四、接口一览（三个后端一致）

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| POST | `/auth/register` | 注册（返回 token） | 否 |
| POST | `/auth/login` | 登录（返回 token） | 否 |
| GET | `/auth/me` | 当前用户 | 是 |
| GET | `/conversations` | 我的会话列表（不含已删除） | 是 |
| GET | `/conversations/{id}/messages` | 某会话全部消息 | 是 |
| PATCH | `/conversations/{id}` | 重命名 `{title}` | 是 |
| DELETE | `/conversations/{id}` | 逻辑删除 | 是 |
| POST | `/chat/stream` | 流式对话（登录则落库 + DB 上下文） | 可选 |

## 五、三种实现怎么落地

数据访问各用各生态的主流方案，业务一致：

| | LangChain Python | LangChain4j | Spring AI |
|---|---|---|---|
| 数据访问 | SQLAlchemy + PyMySQL | Spring Data JPA | Spring Data JPA |
| 密码哈希 | `bcrypt` | spring-security-crypto `BCrypt` | 同左 |
| JWT | PyJWT | jjwt 0.12 | jjwt 0.12 |
| 落库时机 | SSE 生成器结束时 | `TokenStream.onCompleteResponse` | `Flux...doOnNext` 累加 + `Flux.defer` 收尾 |

### 多轮上下文 = 数据库历史（登录用户）

这是关键一环：登录用户的对话上下文**从数据库取最近 N 条**，而不是只靠进程内记忆。
三个框架记忆机制不同，所以做法不同，但效果一致（重启也能续上下文）：

- **Python**：直接从 DB 查最近 N 条，构造成 history 塞进 prompt。
  ```python
  history = load_db_history(conv_id, user_id, DB_CONTEXT_MESSAGES)   # 登录
  # 匿名才用进程内 CONVERSATIONS 窗口
  ```
- **LangChain4j**：登录态绕过 AiService 的进程内记忆，直接调底层 `streamingChatModel`，
  把 DB 历史拼成消息列表喂进去。
  ```java
  List<ChatMessage> msgs = new ArrayList<>();
  msgs.add(SystemMessage.from("你是一个" + role + "……"));
  for (Message m : history.recentMessages(userId, convId, historyMessages))
      msgs.add("user".equals(m.getRole()) ? UserMessage.from(m.getContent()) : AiMessage.from(m.getContent()));
  msgs.add(UserMessage.from(req.message()));
  streamingChatModel.chat(msgs, handler);
  ```
- **Spring AI**：把 DB 历史"回种"进进程内 `ChatMemory`（仅当为空，比如刚重启），
  之后记忆 Advisor 照常用，等于带上了历史。
  ```java
  if (userId != null && chatMemory.get(convId).isEmpty()) {
      List<Message> seed = ...; // 从 DB 取，转成 Spring AI 的 UserMessage/AssistantMessage
      if (!seed.isEmpty()) chatMemory.add(convId, seed);
  }
  ```

### ⚠️ Token 上限（重要）

历史**不是全量喂**，只带最近 `CHAT_HISTORY_MESSAGES` 条（`.env`，默认 **10** 条 ≈ 5 轮），控制 token。

- 知识库的核心查询接口 **`/ask`（RAG）和 `/agent`（Agent）本来就不带历史**（每次独立检索/回答），
  所以它们不消耗历史 token；只有纯对话 `/chat/stream` 才带历史，且已封顶。
- 想更省/更连贯就调 `CHAT_HISTORY_MESSAGES`。更精细的"按 token 预算截断 / 滚动摘要"是后续可选项。

## 六、前端（单页 React）

- 左下角**登录/注册**弹窗；登录后存 token（localStorage），请求自动带 `Authorization`。
- 会话列表：**登录 → 云端**（`GET /conversations`，点开懒加载消息）；**匿名 → 本地**（localStorage）。
- 历史项支持**双击/✎ 重命名**、**✕ 删除**（登录时调服务端，逻辑删除）。
- 会话 id 直接作为 conversation_id，前后端一致；请求体同时带 `conversation_id`(蛇形) 和
  `conversationId`(驼峰)，三个后端各取所需。

## 七、配置（项目根 `.env`）

```bash
# MySQL
DB_HOST=127.0.0.1
DB_PORT=3306
DB_USER=root
DB_PASSWORD=root
DB_NAME=knowledge_assistant
# JWT（三后端共用，token 才互认；生产换成长随机串）
JWT_SECRET=...
# 多轮上下文带入的历史条数上限
CHAT_HISTORY_MESSAGES=10
```

## 八、验证 checklist

```bash
# 注册 → 拿 token
curl -X POST http://localhost:8001/auth/register -H "Content-Type: application/json" -d '{"username":"demo","password":"secret123"}'
# 带 token 连续两轮（第二轮应记得第一轮 → 上下文来自 DB）
curl -N -X POST http://localhost:8001/chat/stream -H "Content-Type: application/json" -H "Authorization: Bearer <token>" -d '{"message":"记住我叫小明","conversation_id":"<uuid>"}'
curl -N -X POST http://localhost:8001/chat/stream -H "Content-Type: application/json" -H "Authorization: Bearer <token>" -d '{"message":"我叫什么？","conversation_id":"<uuid>"}'
# 列表 / 重命名 / 删除
curl http://localhost:8001/conversations -H "Authorization: Bearer <token>"
curl -X PATCH http://localhost:8001/conversations/<uuid> -H "Content-Type: application/json" -H "Authorization: Bearer <token>" -d '{"title":"新标题"}'
curl -X DELETE http://localhost:8001/conversations/<uuid> -H "Authorization: Bearer <token>"
```

> 8001=Python / 8080=LangChain4j / 8082=Spring AI。三个后端同样的接口、同一套账号。

- [ ] 三个后端都能注册/登录、未带 token 访问 `/conversations` 返回 401
- [ ] 带 token 连续两轮，第二轮能记住第一轮的内容（DB 上下文生效）
- [ ] 重命名后列表标题更新；删除后列表不显示，但 DB 里 `status=1`（行还在）
- [ ] 用 Python 注册的账号，去 LangChain4j / Spring AI 也能用同一个 token
