# RAG（检索增强生成）基础

RAG = Retrieval-Augmented Generation，"检索增强生成"。它是把外部知识库引入
大语言模型应用的最主流方案，让模型回答用户问题时**先查资料、再开口**。

## 为什么需要 RAG

LLM 自身有三个硬限制：

1. **知识截止日期**：训练完成后发生的事它都不知道。
2. **幻觉**：对不知道的问题，模型经常自信地编造答案。
3. **私有数据无知**：你公司的内部文档、产品手册它从没见过。

RAG 通过"先检索相关文档，再让模型基于这些文档回答"绕开这些问题。

## RAG 标准流程

整个流程分两个阶段：

### 阶段一：索引（Indexing，离线）

1. **加载（Load）**：从 PDF、Markdown、网页、数据库等读取原始文档。
2. **切块（Split）**：把长文档切成几百字一块的"chunk"，通常保留 10%-20% 重叠。
3. **嵌入（Embed）**：用 embedding 模型把每个 chunk 变成 768 或 1024 维的向量。
4. **存储（Store）**：把向量 + 原文 + 元数据存进向量库（Chroma、Milvus 等）。

### 阶段二：查询（Query，在线）

1. **嵌入问题**：用同一个 embedding 模型把用户问题变成向量。
2. **相似度检索**：在向量库里找语义最相近的 Top K 个 chunk（通常 3-5 个）。
3. **拼接 prompt**：把检索到的 chunk 作为 context 塞进 prompt 模板。
4. **生成回答**：LLM 基于 context 回答，并可以引用具体来源。

## 关键调参点

- **chunk_size**：太小信息不完整，太大稀释关键信号。中文一般 300-800 字符。
- **chunk_overlap**：跨块边界保留上下文，常用 10%-20% 的 chunk_size。
- **Top K**：检索几条。少了召回不够，多了 prompt 太长且引入噪声。一般 3-5。
- **相似度阈值**：低于阈值就当作"知识库里没有"，可以让模型坦白说不知道。

## RAG 不是银弹

RAG 解决"找到相关原文"的问题，但解决不了：

- **需要跨文档推理的复杂问题**——这需要 Agent + 工具调用。
- **检索质量本身**——embedding 模型选不对、切块策略不当，RAG 一样会瞎答。
- **文档不存在的问题**——模型仍然可能幻觉。需要 prompt 工程明确要求"找不到就说不知道"。

进阶方案包括 HyDE（用 LLM 改写问题再检索）、RAG-Fusion（多路检索融合）、
Re-ranker（用更强的模型重排候选）等。
